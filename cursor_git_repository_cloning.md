## 1) 이 프로젝트의 주제(무엇을 하려는가)

핵심 주제는 다음 한 줄로 요약됩니다.

- **현실의 씨앗 정보(뉴스/정책/리포트/소설 등)를 입력하면, 그 정보를 바탕으로 다수 에이전트 사회를 구성해 시뮬레이션하고, 미래 전개를 예측 보고서로 반환한다.**

즉,
- 검색/요약 중심 AI가 아니라
- **“개체(에이전트) 상호작용에서 나타나는 집단 창발(emergence)”**을 이용해
- 정책/여론/금융/스토리 결말 같은 “what-if” 시나리오를 탐색하는 엔진입니다.

## 2) 작동 원리(파이프라인)

README의 5단계 워크플로우가 실제 백엔드 API 구조와 거의 1:1로 대응됩니다.

1. **Graph Build 단계**
   - 사용자가 문서 업로드 + 시뮬레이션 요구사항 입력
   - 문서 텍스트 추출/전처리
   - LLM이 **Ontology(엔티티 타입/관계 타입)** 생성
   - 텍스트를 청크로 나눠 Zep 그래프 메모리에 적재
   - 결과: `graph_id`, 노드/엣지 구조 확보

2. **Environment Setup 단계**
   - 그래프에서 엔티티(인물/집단/기관 등) 필터링
   - 각 엔티티를 OASIS 에이전트 프로필로 변환
   - LLM이 시뮬레이션 설정(시간축, 라운드, 이벤트, 플랫폼 파라미터)을 자동 생성
   - 결과: 시뮬레이션 준비 파일(`profiles`, `simulation_config`) 완성

3. **Simulation 단계**
   - Twitter/Reddit 두 플랫폼을 병렬 또는 선택 실행
   - 라운드별로 에이전트 행동(포스트/댓글/반응) 생성
   - 실행 중 상태, 타임라인, 액션 로그, DB 기반 콘텐츠 조회 가능
   - 옵션으로 행동 로그를 그래프 메모리에 다시 반영 가능(동적 메모리 업데이트)

4. **Report Generation 단계**
   - ReportAgent가 시뮬레이션 결과 + 그래프를 도구적으로 탐색
   - 섹션 단위로 보고서를 비동기 생성
   - 진행률/중간 섹션/최종 Markdown 다운로드 제공

5. **Deep Interaction 단계**
   - 시뮬레이션 속 특정 에이전트 인터뷰
   - 배치 인터뷰/전체 인터뷰
   - ReportAgent와 질의응답(chat)으로 결과 해석

## 3) 구조적 특징(아키텍처 관점)

### A. 프론트/백 분리형
- `frontend`는 Vue 기반 step UI (`Step1~Step5`)로 파이프라인 유도
- `backend`는 Flask API로 단계별 비동기 작업 수행

### B. API 도메인 분리
- `/api/graph`: 프로젝트/온톨로지/그래프 구축/태스크 추적
- `/api/simulation`: 엔티티 조회, 준비, 실행 제어, 실시간 상태, 인터뷰
- `/api/report`: 보고서 생성/진행률/조회/다운로드/에이전트 채팅

### C. 비동기 태스크 중심 설계
- 그래프 구축, 시뮬 준비, 보고서 생성은 모두 장시간 작업
- 즉시 `task_id`를 반환하고 상태 폴링으로 진행률 확인
- 실제 서비스 사용성을 고려한 구조

### D. 외부 엔진 결합
- **Zep**: 그래프 메모리/지식 구조 저장 및 조회
- **OASIS**: 멀티 에이전트 사회 시뮬레이션 실행 엔진
- **LLM(OpenAI SDK 호환)**: 온톨로지 생성, 프로필/설정 생성, 보고서 작성

### E. 결과물 2트랙
- 정적 결과: 상세 예측 보고서(Markdown)
- 동적 결과: 상호작용 가능한 시뮬레이션 세계(에이전트 인터뷰/행동 로그)

## 4) 이 프로젝트의 “진짜 차별점”

- **문서 업로드 → 그래프화 → 에이전트 사회화 → 병렬 시뮬레이션 → 보고서 + 대화형 분석**까지 한 파이프라인으로 묶음
- 단순 텍스트 추론이 아니라 **행동 시뮬레이션 기반 예측**
- “거시적 정책 실험”과 “미시적 창작/스토리 실험” 두 유스케이스를 동시에 겨냥
- 실험 반복을 위한 운영 기능(히스토리, 상태조회, 재실행, 강제 재생성, 다운로드 API)이 꽤 잘 갖춰져 있음

## 결론 먼저
- **직접 매매/리스크관리/알파 추구** 목적이면: **박사논문 모델이 더 적합**
- **사회여론·정책반응·집단행동 시나리오 탐색** 목적이면: **MiroFish가 더 적합**
- 실무적으로는 **둘을 결합**하는 게 가장 강력합니다.

## 왜 박사논문 쪽이 금융투자에 더 유리한가
PDF 초안은 목적 자체가 금융투자에 맞춰져 있습니다.
- System1(초고속 실행) + System2(느린 계획) + FiLM 제어
- 목표 지표가 명확히 투자지표 중심(CAGR, Sharpe, Sortino, MDD)
- 위기구간 대응(2008, 2020, 2022), 슬리피지/지연시간까지 다룸
- “예측”보다 “리스크 방어 + 생존 전략”에 초점

즉, **매매 실행 체계**(execution, latency, risk budget)까지 고려한다는 점에서 투자용에 더 직접적입니다.

## 왜 MiroFish는 성격이 다른가
MiroFish는 멀티 에이전트 기반의 **사회 시뮬레이션 엔진**입니다.
- 강점: 군중심리, 이슈 확산, 정책 반응, 내러티브 전이
- 산출물: 예측 리포트 + 상호작용 가능한 디지털 사회

## 왜 이 조합이 “한계 극복”에 맞나

- **신뢰성 부족(별/릴리즈/운영성)** → 대형 커뮤니티 프로젝트로 대체
- **백오피스 부재** → ERPNext/Odoo로 재무·운영 체계 확보
- **트레이딩 리스크 통제 부족** → Freqtrade/Lean + Temporal로 실행 안정성 강화
- **감사 가능성 부족** → Langfuse로 추적/감사/품질관리
- **업무 연결성 부족** → n8n으로 실무 툴(슬랙/메일/DB/CRM) 연결

## 2) 서비스별 대략 리소스/용량

- `LangGraph`(에이전트 오케스트레이션)
  - RAM 2~8GB, CPU 1~4코어
  - 로그/상태 저장: 월 10~100GB

- `n8n`(자동화 허브)
  - RAM 1~4GB, CPU 1~2코어
  - 워크플로우/실행로그: 월 5~50GB

- `ERPNext` 또는 `Odoo`(백오피스)
  - RAM 4~16GB, CPU 2~6코어
  - DB + 첨부파일: 월 10~200GB (회사 사용량 따라 큼)

- `Freqtrade`/`Lean`(자동매매)
  - RAM 2~8GB, CPU 2~4코어
  - 시세데이터/백테스트결과: 월 20~300GB

- `Langfuse`(관측/감사)
  - RAM 4~12GB, CPU 2~6코어
  - 트레이스/로그 폭증 가능: 월 50~500GB

- `Temporal`(워크플로우 내구성)
  - RAM 4~12GB, CPU 2~6코어
  - 이벤트 히스토리 축적: 월 20~200GB

- `Qdrant + Haystack`(지식/RAG)
  - RAM 8~32GB (데이터량 따라 급증)
  - 벡터 인덱스: 문서 100만 chunk 기준 수백 GB 가능
